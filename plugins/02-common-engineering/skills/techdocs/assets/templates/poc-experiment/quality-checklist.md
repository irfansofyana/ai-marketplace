# POC/Technical Experiment Quality Validation Checklist

⚠️ **VALIDATION PHASE ONLY**: Use this checklist AFTER generating the complete draft, before presenting to the user.

⚠️ **DO NOT include this checklist in the user-facing document.**

⚠️ **DO NOT include checkboxes (- [ ]) in the user's document.** This is for your internal validation only.

---

## How to Use This Checklist

**CORRECT workflow:**
1. ✅ Generate complete draft WITHOUT any checkboxes
2. ✅ INTERNALLY validate using this checklist
3. ✅ Present clean draft to user
4. ✅ Offer improvements based on validation (without showing checklist)

**WRONG workflow:**
1. ❌ Include this checklist in the user's document
2. ❌ Show checkboxes to the user
3. ❌ Present validation criteria as part of the draft

---

## Validation Criteria

### Completeness

- [ ] All required sections are filled out
- [ ] Metadata (author, status, dates) is complete
- [ ] Success criteria are defined with targets
- [ ] Go/No-Go decision is clearly stated

### Clarity

- [ ] Problem statement is clear and specific
- [ ] Success criteria are measurable
- [ ] Approach is understandable
- [ ] Results are quantified where possible
- [ ] Recommendation is supported by evidence

### Experimental Rigor

- [ ] Success criteria were defined upfront
- [ ] Results are compared to targets
- [ ] Evidence supports conclusions
- [ ] Limitations are acknowledged
- [ ] Trade-offs are documented

### Decision Quality

- [ ] Go/No-Go decision is clear
- [ ] Decision rationale connects to results
- [ ] Production readiness needs are documented (if Go)
- [ ] Alternative suggestions are provided (if No-Go)

### Format

- [ ] Follows template structure exactly
- [ ] Success criteria table is complete
- [ ] Metrics table includes baseline and change
- [ ] Appendix includes relevant links

### Content Quality (Per Section)

**Executive Summary:**
- [ ] Complete in 3-5 sentences
- [ ] Includes the decision (Go/No-Go)
- [ ] Quantified findings if applicable
- [ ] Stands alone (reader understands the conclusion)

**Problem Statement:**
- [ ] Clear problem or question stated
- [ ] Pain points are specific
- [ ] Success criteria are measurable
- [ ] Target values are defined

**Approach & Solution Design:**
- [ ] Rationale explains why this approach
- [ ] Alternatives were considered
- [ ] Key technologies have clear purpose
- [ ] Scope boundaries are explicit

**Implementation Details:**
- [ ] Steps are clear and reproducible
- [ ] Commands/configurations included
- [ ] Deviations from plan documented

**Results & Findings:**
- [ ] Success criteria results are evaluated
- [ ] Metrics include baseline and change
- [ ] Evidence is included where possible
- [ ] Unexpected discoveries are noted

**Trade-offs & Limitations:**
- [ ] Trade-offs are acknowledged
- [ ] Limitations are documented
- [ ] Technical debt is noted
- [ ] Honest assessment provided

**Recommendation & Next Steps:**
- [ ] Clear decision stated upfront
- [ ] Rationale connects to results
- [ ] Production readiness needs assessed (if Go)
- [ ] Alternative suggested (if No-Go)
- [ ] Next steps are actionable

**Appendix:**
- [ ] Links to repos/branches provided
- [ ] Raw data/logs referenced
- [ ] Relevant documentation cited

---

## Post-Validation Actions

After validating internally:

1. **If all criteria met**: Present the draft to user as-is
2. **If gaps identified**:
   - Ask targeted questions to fill gaps
   - Generate additional content
   - Re-validate
3. **Offer improvements**: Suggest enhancements without showing this checklist
4. **User refinements**: Let user request changes
5. **Final check**: Ensure no prohibited content (checkboxes, HTML comments, validation criteria) appears in final output

---

## Pre-Output Self-Check

Before presenting ANY draft to the user, verify:

- [ ] ❌ NO checkboxes (✅ or - [ ]) in the document
- [ ] ❌ NO HTML comments in the document
- [ ] ❌ NO quality criteria text in the document
- [ ] ❌ NO "GUIDANCE QUESTIONS" sections in the document
- [ ] ❌ NO references to this validation checklist in the document
- [ ] ✅ ONLY section headings and user-provided content

**If ANY prohibited content found**: Remove it immediately before presenting to user.
